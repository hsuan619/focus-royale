
## 專注力大逃殺 (Focus Royale) 實作說明書

### 一、 最佳市場切入點：「手機監牢」模式

現代使用者習慣多螢幕作業，純網頁 MVP 無法干涉作業系統層級的其他軟體，因此將系統定位由「設備全局鎖定」轉變為「針對性實體封鎖」，創造出零摩擦力的市場切入點。

* **使用情境定義**：
1. **生產力分流**：使用者使用平板或筆記型電腦作為主要讀書、工作設備，系統完全不干涉其筆記軟體或線上課程的運作。
2. **物理性隔離**：使用者必須使用最大的分心來源——「智慧型手機」，開啟瀏覽器進入本遊戲網頁，並將手機放置於桌面上。
3. **絆線觸發**：遊戲期間手機螢幕必須維持恆亮且停留在該網頁。一旦使用者忍不住切換應用程式（如回覆訊息），網頁退至背景即刻觸發淘汰機制。


* **商業優勢**：無需使用者下載安裝任何原生 App，也無需索取敏感的作業系統監控權限。只要點擊網址輸入暱稱即可參戰，大幅降低獲客門檻。

### 二、 核心系統與技術架構

系統捨棄複雜的 3D 渲染與物理引擎，以純粹的狀態監聽與資料庫互動構成。

#### 1. 狀態判定演算法 (Page Visibility API)

判定淘汰的絕對條件（手機端）：
只要手機瀏覽器觸發 visibilitychange 事件，且 document.hidden === true，系統立刻發送淘汰封包。

會觸發淘汰的手機日常行為：

切換 App：玩家點開了螢幕上方的 Line 訊息通知，或是上滑切換到 Instagram。

退回桌面：玩家按下 Home 鍵或底部上滑，回到手機桌面。

螢幕休眠：玩家按下了電源鍵關閉螢幕（代表他們沒有在監控自己的專注狀態）。

系統級覆蓋：玩家重度下拉了 iOS 的「控制中心」或 Android 的「通知面板」並停留超過極短的緩衝時間。

1. 嚴格判定：視窗背景化 (Backgrounding)
這是核心防線。只要瀏覽器不再是使用者當前正在觀看的唯一畫面，即刻淘汰。

邏輯：if (document.visibilityState === 'hidden') { triggerElimination(); }

2. 寬容判定：系統中斷處理 (System Interruptions)
手機與電腦不同，手機會有「無法抗拒的系統中斷」，例如突然有人打電話來、或是跳出「電量低於 20%」的系統警告。這些狀況會觸發手機網頁的 window.onblur（失去焦點），但此時網頁可能還沒被完全隱藏。

修正後的防寬容機制：當觸發 window.onblur 時，不直接淘汰，而是啟動 3 秒至 5 秒的緩衝倒數（畫面上可顯示：「偵測到系統中斷，請於 5 秒內返回遊戲」）。

若玩家只是按掉低電量警告，馬上點回網頁，倒數取消，遊戲繼續。

若玩家接起電話或點開了通知，導致網頁進一步觸發了 hidden 狀態，則毫無懸念，立刻淘汰。

#### 2. 積分與獎勵結算數學模型 (Scoring Algorithm)

系統依據存活時間與同房競爭者的淘汰狀況進行積分結算：


$$ S_{final} = \left\lfloor t \times B \right\rfloor \times \left( 1 + \frac{N_{eliminated}}{N_{total}} \times \alpha \right) $$

* $S_{final}$：玩家最終獲得的總積分。
* $t$：有效專注存活時間（分鐘）。
* $B$：基礎時間常數（例如每專注 1 分鐘獲得 10 分，$B = 10$）。
* $N_{total}$：開局時房間內的總人數。
* $N_{eliminated}$：玩家存活期間，房間內被淘汰的總人數。
* $\alpha$：大逃殺難度加成係數（設定介於 $0.5$ 至 $1.5$ 之間），人數減少越多，倖存者獲得的積分乘數越高。

### 三、 遊戲機制與市場差異化設計

傳統鎖機軟體依賴系統權限進行「阻斷」，本專案則利用心理制約與社會連動創造「不願退出」的誘因。

* **損失厭惡與押注機制**：
導入行為經濟學的損失厭惡模型：

$$ U(x) = \begin{cases} \lambda x & \text{if } x < 0 \\ x & \text{if } x \ge 0 \end{cases} $$



進階房間要求扣除虛擬金幣作為押金（即 $x < 0$ 的情境）。由於 $\lambda > 1$（通常為 2.25），玩家對失去押金的恐懼會轉化為強大的專注力。倖存者將平分淘汰者的押金池（零和博弈）。
* **戰況實時推播（WebSocket）**：
伺服器實時推播他人失敗的訊息（例如：「玩家 A 已遭到淘汰。剩餘存活：42/100」）。他人的淘汰直接意味著自身預期收益的增加，強化競技虛榮感。
* **白噪音與沉浸介面**：
提供純粹的環境音（雨聲、營火等），並強制採用極簡的深色模式（Dark Mode），確保其生產力工具的專業屬性不被過度遊戲化的介面干擾。

### 四、 螢幕常亮的耗電問題與技術解法

「手機監牢」要求螢幕恆亮，為避免電池快速耗盡，網頁前端必須嚴格執行以下硬體最佳化策略：

#### 1. OLED 物理省電特性結合 (OLED Power Efficiency)

智慧型手機的顯示器總功耗 $P_{display}$ 公式為：


$$ P_{display} \approx P_{base} + c \cdot \sum_{i=1}^{N} (R_i + G_i + B_i) $$


（$P_{base}$ 為基礎靜態功耗，$c$ 為發光效率常數，$N$ 為像素總數，$R_i, G_i, B_i$ 為色彩亮度值）。

* **技術解法**：UI 介面採用 **True Black（純黑背景，Hex: #000000）**。在 OLED 螢幕上，純黑像素為物理性斷電不發光，這能將絕大多數像素的 RGB 值歸零，使 $P_{display}$ 降至接近底層基礎功耗。

#### 2. 螢幕喚醒鎖定與亮度控制 (Screen Wake Lock API)

* **技術解法**：於客戶端載入時呼叫 navigator.wakeLock.request('screen')，強制覆蓋作業系統的螢幕休眠機制。同時在畫面上常駐文字，引導使用者手動將手機實體亮度調至最低。

#### 3. 網路封包與通訊節流 (Heartbeat Optimization)

持續的高頻網路傳輸會喚醒無線通訊模組，造成顯著耗電。

* **技術解法**：拉長 WebSocket 的心跳封包與戰況同步間隔（例如設定為 $\Delta t = 10$ 秒）。當使用者觸發 visibilitychange 違規時，才以非同步的 HTTP POST 即刻發送淘汰訊號，以最低頻率佔用網路資源。

### 五、 技術選型

#### 前端（Client）

| 技術 | 選型 | 理由 |
|------|------|------|
| 語言/框架 | Vanilla HTML / CSS / JavaScript | 無框架載入負擔，手機首屏速度最快；本專案互動邏輯簡單，不需 React/Vue |
| 樣式 | Tailwind CSS (CDN) | 快速實現 True Black 深色介面，無需自訂 CSS 架構 |
| 即時通訊 | Socket.io Client | 內建自動重連機制，處理手機網路抖動優於原生 `ws` |
| 螢幕常亮 | Screen Wake Lock API + `noSleep.js` 降級 | 原生 API 優先；iOS Safari 不支援時以 `noSleep.js`（無聲影片循環）作 fallback |
| 環境音 | Howler.js | 輕量音訊庫，支援背景音循環與手機自動播放限制的解法 |

#### 後端（Server）

| 技術 | 選型 | 理由 |
|------|------|------|
| 執行環境 | Node.js 20 LTS | 事件驅動模型天然契合 WebSocket 高並發連線 |
| HTTP 框架 | Fastify | 比 Express 吞吐量高 2–3 倍；Schema 驗證內建 |
| WebSocket | Socket.io Server | 與前端對應；房間 (Room) 抽象直接對應遊戲房間概念 |
| 排程/定時器 | Node.js 原生 `setInterval` | 僅用於房間倒數開局與心跳，不需額外排程庫 |

#### 資料層（Data）

| 技術 | 選型 | 用途 |
|------|------|------|
| PostgreSQL | 主資料庫 | 用戶帳號、積分歷史、排行榜（持久化資料） |
| Redis | 即時狀態快取 | 房間狀態、在線玩家清單、押注池（揮發性高頻讀寫） |
| Prisma | ORM | PostgreSQL Schema 管理與類型安全查詢 |

#### 部署（DevOps）

| 技術 | 選型 | 理由 |
|------|------|------|
| 後端部署 | Railway | 支援 Node.js + Redis + PostgreSQL 整合，免設定 WebSocket |
| 前端部署 | Vercel 或直接由 Fastify 靜態服務 | MVP 階段後者可減少跨域設定複雜度 |
| 環境變數管理 | `.env` + Railway 環境變數面板 | — |

#### 已確認架構決策

1. **用戶身份**：採用 **Google OAuth** 綁定，支援積分歷史與跨裝置排行榜。前端使用 Google Identity Services (GIS)，後端驗證 ID Token 後發放 JWT session。
2. **房間開局條件**：最少 **2 人**加入後啟動 **60 秒倒數**，時間到自動開局；倒數中若有新玩家加入，倒數不重置（避免無限等待）。
3. **斷線判定**：WebSocket 斷線後給予 **10 秒**重連緩衝（獨立於 blur 的 5 秒緩衝）。10 秒內重連成功則遊戲繼續；逾時視同主動離開，觸發淘汰。
4. **α 係數**：依房間模式固定三檔——普通房 α=0.5、進階押注房 α=1.0、錦標賽房 α=1.5。
5. **金幣初始來源**：新用戶首次登入贈送 **100 金幣**；每日首次登入 +10；存活獎勵依積分公式另計（不直接給幣）。
